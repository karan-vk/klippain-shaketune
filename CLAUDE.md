# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Shake&Tune (`shaketune`) is a Klipper plugin, part of the Klippain ecosystem, that adds G-code macros and a companion CLI for input-shaper/resonance calibration on 3D printers. It records accelerometer data during test moves and renders matplotlib graphs (belts comparison, input shaper, vibrations profile, axes map, static frequency) to help diagnose mechanical issues. It runs as a Klipper "extra" (`[shaketune]` in `printer.cfg`), i.e. inside `klippy`, the single-threaded Python process that also drives real-time step generation — this constraint (don't block the Klipper reactor) shapes most of the architecture below.

The package is not published to PyPI; it's deployed by `install.sh`, which clones this repo to `~/klippain_shaketune` and symlinks `shaketune/` into `<klipper>/klippy/extras/shaketune`.

## Commands

- Lint: `ruff check` (config in `pyproject.toml`: line-length 120, single quotes, target py39). CI runs exactly this in the `lint` job.
- Format: `ruff format`.
- There is no Python unit test suite (no pytest). Correctness is verified by a Klipper integration smoke test (`.github/workflows/test.yml`, `klippy_testing` job) run against a real checkout of `klipper3d/klipper` and `KalicoCrew/kalico` (both `master` and a pinned release, across Python 3.9/3.11). To reproduce locally: clone Klipper, symlink this repo's `shaketune/` into `<klipper>/klippy/extras/shaketune`, then run `klippy.py --import-test` and `scripts/test_klippy.py -d <dict-dir> ci/smoke-test/klippy-tests/simple.test` (a minimal `printer.cfg` + G-code sequence in `ci/smoke-test/klippy-tests/simple.cfg`/`.test` that just loads `[shaketune]` with an ADXL345 + MPU9250 and a `[resonance_tester]` section).
- CLI (generates a graph from previously-recorded data without a running Klipper): `python -m shaketune.cli {static_freq,axes_map,belts,input_shaper,vibrations} -o out.png data.stdata [options]`. Useful for iterating on plotting/computation code without a printer — see `shaketune/cli.py` for per-subcommand flags (`belts`/`input_shaper`/`vibrations` accept `-k/--klipper_dir` to reuse Klipper's own `shaper_calibrate`/`shaper_defs` modules for the math).

## Architecture

### Klipper integration & process model

- `shaketune/__init__.py` exposes `load_config(config)`, the standard Klipper extras entry point; it's called once when `printer.cfg` has a `[shaketune]` section and constructs `ShakeTune` (`shaketune.py`).
- `ShakeTune.__init__` reads settings into a single `ShakeTuneConfig` (`shaketune_config.py` — paths, `keep_raw_data`, `chunk_size`, `max_freq`/`max_freq_vibrations`, `dpi`, etc.), registers a `klippy:connect` handler that validates `[resonance_tester]` is configured (deferred because config load order isn't guaranteed at `__init__` time), and registers the 5 G-code commands.
- If `show_macros_in_webui` (default true), the real commands are registered with a leading underscore and Mainsail/Fluidd-visible `gcode_macro` wrappers are injected directly into the live config object's `fileconfig` — an intentional, acknowledged-in-comments hack against Klipper internals, and a recurring source of breakage when Klipper changes config-loading internals (see recent commits patching Klipper's December 2025 breaking change).
- **Heavy work (numpy/scipy/matplotlib) never runs on Klipper's main thread.** Each command handler does the real-time part synchronously (toolhead moves, accelerometer recording) then hands off to `ShakeTuneProcess` (`shaketune_process.py`), which spawns a `multiprocessing.Process` (explicitly not a thread — comments cite GIL contention causing CANbus issues) to load the recorded data and call `graph_creator.create_graph(...)`. The parent waits via a `reactor.pause()` polling loop (`wait_for_completion`), not `process.join()`, so Klipper keeps servicing timers/events. The child process lowers its own scheduling priority (`SCHED_BATCH`) and has an independent `threading.Timer`-based hard timeout (`os._exit(1)`) so timeout enforcement doesn't depend on the parent ever resuming. Only plain/picklable data (config values, file paths, the graph creator, a measurements manager) crosses the process boundary — never live `printer`/`toolhead`/`reactor` objects.
- `cli.py` (`python -m shaketune.cli`) reuses the exact same `GraphCreatorFactory`/`MeasurementsManager`/`ShakeTuneConfig` classes standalone, with a config built from CLI defaults and no Klipper reactor (`k_reactor=None`), bypassing `ShakeTuneProcess` entirely since there's no reactor to protect.

### Command layer (`shaketune/commands/*.py`)

One module per G-code macro (`AXES_MAP_CALIBRATION`, `AXES_SHAPER_CALIBRATION`, `COMPARE_BELTS_RESPONSES`, `CREATE_VIBRATIONS_PROFILE`, `EXCITATE_AXIS_AT_FREQ`), all following the same skeleton:

1. Save current toolhead velocity/accel limits and disable `input_shaper` (except `create_vibrations_profile`, which *requires* input shaping to already be configured and deliberately leaves it enabled, since it measures the machine under normal shaped motion).
2. Move to a start point, then per axis/segment: start accelerometer recording into a `MeasurementsManager`, drive the toolhead (either a raw move, or a frequency-swept/fixed pulse train from `helpers/resonance_test.py`), stop recording.
3. Restore the saved velocity/accel limits and input shaper state.
4. Call `graph_creator.configure(...)` with run-specific parameters, `measurements_manager.save_stdata()`, then `st_process.run(filename)` + `wait_for_completion()`.

Some patterns are duplicated across all 5 files rather than centralized (worth knowing before "fixing" just one copy): the `minimum_cruise_ratio`-vs-legacy `SET_VELOCITY_LIMIT` branch, the single-`test_point`-required validation, and empty-string-to-`None` coercion for gcode params from templated macros. `axes_shaper_calibration` runs the whole record→configure→run→wait cycle once *per axis* (separate output files), whereas `compare_belts_responses`/`axes_map_calibration` record everything into one shared file and only spawn the graph-creation subprocess once.

### Graph creation: factory + registry, computation/plotting composition

Each of the 5 graph types is wired together from three pieces, joined by a shared `graph_type` string (e.g. `'belts comparison'`, `'input shaper'`):

- **`GraphCreator`** subclasses (`shaketune/graph_creators/*_graph_creator.py`) register themselves globally via the `@GraphCreator.register('type name')` class decorator (`graph_creator.py`), which populates `GraphCreator.registry`. `GraphCreatorFactory.create_graph_creator(graph_type, config)` just looks up `registry[graph_type]` and instantiates it — adding a new graph type means adding a new subclass decorated with `@register(...)` (imported somewhere so the decorator runs), *not* touching the factory. Each subclass's `__init__(self, config)` hardcodes which `Computation`/`PlotterStrategy` pair it uses via `super().__init__(config, ComputationCls, PlotterCls)`. `GraphCreator.create_graph()` itself is a fixed, non-overridable pipeline: `computation.compute()` → `plotter.plot(result)` → `self._save_figure(fig)` (this is also where `config.dpi` is applied — nowhere else).
- **Computation classes** (`graph_creators/computations/*.py`) are plain numpy/scipy classes (no shared base class — they satisfy a structural `Protocol` in `base_models.py` just by having `compute()`) that turn raw `Measurement` samples into a typed, immutable `ComputationResult` dataclass (`computation_results.py`: `BeltsResult`, `ShaperResult`, `VibrationsResult`, `AxesMapResult`, `StaticFrequencyResult`). `get_plot_data()` flattens a result into the plain dict that is the *only* thing crossing into plotting code.
- **Plotter classes** (`graph_creators/plotters/*.py`) subclass `PlotterStrategy` (`base_models.py`) and turn that dict into a matplotlib `Figure`, using shared chrome helpers (`add_logo`, `add_title`, `add_version_text`) and layout/styling helpers in `plotting_utils.py` (`AxesConfiguration.configure_axes`, `SpectrogramHelper`, `TableHelper`, the `KLIPPAIN_COLORS` brand palette — note this palette is duplicated between `plotting_utils.py` and `base_models.py`, so update both).
- This computation/plotting split is a **code-organization boundary only**, not a process boundary — both run inside the same `multiprocessing.Process` spawned by `ShakeTuneProcess`.
- `graph_type` is independently used as a lookup key in at least three places that must be kept in sync by hand when adding a graph type: `GraphCreator.registry`, `ShakeTuneConfig.RESULTS_SUBFOLDERS` (output directory), and `shaketune.py`'s command registration / `cli.py`'s `graph_type_map`.
- Klipper-version compatibility for the shaper-fitting math (`shaper_calibrate.process_accelerometer_data`/`find_best_shaper`) is handled by shims in `graph_creators/__init__.py`, used by `shaper_computation.py`/`belts_computation.py`/`vibrations_computation.py` — this is a **separate compatibility layer from `helpers/compat.py`** (see below); don't conflate the two when debugging a Klipper/Kalico version issue.

### Helpers (`shaketune/helpers/*.py`)

- `accelerometer.py`: `MeasurementsManager` owns the on-disk `.stdata` format (JSON-lines, Zstandard-compressed, written by a dedicated background `multiprocessing.Process`) and can also load legacy Klipper raw-accelerometer CSVs. `Accelerometer` wraps Klipper's internal accelerometer client and reactor to start/stop recordings without blocking Klipper's main thread (`stop_recording` defers the actual finish to a reactor callback, then polls for readiness).
- `resonance_test.py`: reimplements Klipper's resonance-tester motion generation (credited to Dmitry Butyugin) with true Euclidean-length-preserving direction vectors, Z-axis support, and a static-frequency mode; `ResonanceTestManager` drives the toolhead directly through a generated pulse sequence, depending on `helpers/compat.py`'s `KlipperCompatibility` for every version-sensitive toolhead/resonance-tester operation.
- `helpers/compat.py`'s `KlipperCompatibility` hides the fact that Klipper's `resonance_tester` object changed shape in Dec 2024 (legacy `.test.*` sub-object vs. modern `.generator.vibration_generator.*` + `.probe_points`, plus a newer "sweeping" motion mode) and that `toolhead` has gained/lost methods across versions (`set_max_velocities` vs `cmd_M204`, `limit_next_junction_speed`). **It detects capabilities purely via `hasattr()`, never by fork/version identity** — this is deliberate (Kalico and other forks just need to expose an attribute of the right name to be treated correctly) but means a fork that reshapes a same-named attribute is silently misrouted until it fails deeper. This exact failure mode is what recent commits (`c4a8552`, `23c4c3d`) fixed in the *other* compat shim (`graph_creators/__init__.py` + `shaper_computation.py`), by tightening signature/attribute checks — if you're chasing a Kalico/Klipper compatibility bug, check both compat layers, not just `helpers/compat.py`.
- `common_func.py`: shared math (`compute_mechanical_parameters` — half-power-bandwidth resonant frequency/damping ratio, `detect_peaks`, `identify_low_energy_zones`) and `AXIS_CONFIG` (direction vectors for x/y/belt A/belt B/CoreXZ). Prefer numpy-only implementations over scipy here — recent history has actively replaced `scipy.signal`/`scipy.stats` calls with numpy equivalents to reduce footprint on low-power hosts (Raspberry Pi).
- `motors_config_parser.py`: reads TMC driver/motor config and register state (CHOPCONF/PWMCONF/COOLCONF/...) for motor-comparison report tables.
- `console_output.py`: thin print facade so plugin output can be redirected to Klipper's gcode console (`register_output_callback`) instead of stdout.
